# P2 组织与权限体系 — 实现方案

> 基于 Unknowns 报告澄清结果，按 P2.1~P2.7 七个子阶段分步实现。

---

## 总体架构约束

- **Backend**: JDBC/JdbcTemplate、Flyway（mysql + sqlite 双目录同步）、无 JPA/ORM
- **Wiring**: ServiceConfiguration.java @Bean 手动装配
- **错误**: AuthException(errorCode, message) + GlobalExceptionHandler switch
- **测试**: JUnit5 @Nested + stub repository 模式
- **Frontend**: Vue 3 Composition API + `<script setup>` + TypeScript + Pinia + 现有 CSS 工具类
- **路由**: 懒加载 + beforeEach 全局守卫
- **Layout**: 新增 AppLayout.vue（左侧侧边栏 + 右侧 `<router-view>`）

## P2.5：成员角色与授权

### 5A. Authorization Service
- `AuthorizationService.java` 接口 + `AuthorizationServiceImpl.java`
  - `check(userId, orgId, permissionCode)` → AuthorizationResult (ALLOW / DENY_*)
  - `require(userId, orgId, permissionCode)` → 不满足抛异常
  - `getEffectivePermissions(userId, orgId)` → Set<String> 权限码集合
  - 查询链路：Membership → MembershipRole → RolePermission → Permission
  - 组织状态验证：检查 SUSPENDED/DELETED
  - 使用 Caffeine 缓存 (key: userId+orgId, value: permissionCodes+membershipVersion)
  - authorization_version 变化时缓存失效

### 5B. 权限 Snapshot API
- `GET /api/v1/identity/me/organizations/{orgId}/permissions`
  - 返回角色列表 + 权限码集合 + permissionVersion
- 前端用它控制菜单和按钮渲染

### 5C. Internal Authorization API
- `InternalAuthorizationController.java`
  - `POST /internal/v1/identity/authorization/check` — 接受 {userId, organizationId, permission} → {allowed, decision, permissionVersion}

### 5D. 越权防护
- 所有组织资源 API 必须验证目标 `organization_id` 与请求上下文一致
- 角色操作验证角色属于当前组织
- Permission 变更递增 `authorization_version`

### 5E. ServiceConfiguration 注册
- 注册 AuthorizationService, CaffeineCacheManager

### 5F. 前端：权限菜单
- 在 AppLayout 导航中根据 `GET /me/organizations/{id}/permissions` 动态显示/隐藏菜单项
- 按钮级别权限控制（v-if 检查权限码）

### 5G. 测试
- `P2AuthorizationUnitTest.java`: 正常授权、组织冻结拒绝、Membership SUSPENDED 拒绝、多角色权限并集、跨组织权限拒绝、缓存命中+失效、authorization_version 递增验证
