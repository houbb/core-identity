# Core Identity P7：高可用、规模化与生态扩展

版本：P7
前置依赖：P0–P6
目标：将 Core Identity 从企业级身份治理系统升级为可多实例部署、可水平扩展、可灾难恢复、可持续演进并支持安全扩展生态的平台级基础设施。

---

# 一、P7 核心目标

P6 已经解决：

```text
用户为什么拥有权限
谁批准了权限
权限何时失效
权限是否存在职责冲突
访问是否经过周期复核
审计与隐私流程是否可证明
```

P7 继续解决：

```text
单个 Identity 实例故障后系统是否继续运行？
数据库故障后能否安全接管？
多实例之间如何共享 Session 和撤销状态？
大量 Token、审计、事件和 Provisioning 数据如何扩展？
发布新版本时如何不停机？
Schema 升级失败后如何恢复？
区域级故障发生后多久恢复？
Redis、消息队列或下游 Core 不可用时如何降级？
插件如何接入而不破坏身份安全边界？
不同版本的 Core 服务如何保持兼容？
```

```text

单个 Admin Backend

本地文件

```

```text

```

```text
1～2 个 Identity Backend

MySQL 或 PostgreSQL
```


## 2.3 Enterprise HA

```text
多个 Identity Backend
多个 Admin Backend
高可用关系数据库
Redis Sentinel 或 Cluster

对象存储
集中式密钥管理
```

适用：

```text

严格 RTO/RPO
多个 Core 共同依赖 Identity
```

核心原则：

> P7 增加可选企业能力，但不得让社区版必须安装 Redis、Kafka、Kubernetes 和外部 KMS 才能启动。

# 六、无状态多实例

## 6.1 无状态的含义

Identity Backend 实例本身不保存不可恢复的业务事实。

实例本地可以保存：

```text
短期缓存
公钥缓存
配置快照
临时线程池状态
指标
```

实例本地不能作为唯一来源保存：

```text
用户 Session
OAuth Grant
Refresh Token
MFA 注册 Challenge
分布式任务状态
授权版本
Token 撤销状态
```

实例重启后，系统业务结果不能丢失。

---

## 6.2 请求可以落到任意节点

以下流程必须支持跨节点：

```text
Node A 创建登录 Challenge
Node B 完成认证

Node A 签发 Authorization Code
Node C 使用 Code 换 Token

Node B 创建账号恢复
Node A 完成恢复

Node C 接收 SCIM 请求
Node A 继续处理后续同步
```

因此 Challenge、Code、Grant 和 Session 不能只存在本机内存。

---

## 6.3 Sticky Session

P7 不依赖 Sticky Session 保证正确性。

负载均衡器可以启用短期粘滞提高缓存命中，但系统必须在关闭 Sticky Session 后仍然正确运行。

错误设计：

```text
用户只要一直访问 Node A，流程就正常
切换 Node B 后认证失败
```

# 七、数据库演进

## 7.1 数据库支持策略

建议分层支持：

```text
SQLite
    Community Standalone

MySQL
    Standard Production

PostgreSQL
    Enterprise HA 参考实现
```

也可以将 MySQL 作为企业主数据库，但必须为以下能力提供等价实现：

```text
主备复制
自动或受控 Failover
PITR
在线备份
连接池
只读副本
Schema 迁移
```

PostgreSQL 官方文档将高可用、负载均衡、复制、热备、WAL 归档和时间点恢复作为正式能力提供；连续 WAL 归档与基础备份可以用于 PITR。

---

## 7.2 SQLite 到企业数据库迁移

不得让用户手工复制表。

提供正式迁移工具：

```text
scripts/
├── migrate-sqlite-to-mysql
├── migrate-sqlite-to-postgresql
├── verify-migration
└── rollback-migration
```

迁移流程：

```text
进入维护模式
    │
    ▼
停止新写入
    │
    ▼
创建 SQLite 一致性快照
    │
    ▼
导出业务表
    │
    ▼
导入目标数据库
    │
    ▼
校验行数、主键、校验和
    │
    ▼
执行增量补偿
    │
    ▼
切换数据源
    │
    ▼
健康检查
    │
    ▼
解除维护模式
```

小型部署可以停机迁移。

大规模迁移再采用：

```text
双写
变更捕获
短暂停写切换
```

不要在首版就实现长期双写。

---

## 7.3 主库与只读副本

Identity 的关键写操作必须访问主库：

```text
登录失败计数
Session 创建
Token 撤销
权限变更
认证器验证
访问申请
账号恢复
SCIM 更新
```

以下查询可以考虑使用只读副本：

```text
审计报表
访问审查查询
历史登录查询
管理统计
证据生成
```

但必须处理复制延迟。

不能在主库完成权限变更后，立即从延迟副本读取并判断权限仍旧有效。

---

## 7.4 Read-after-write 一致性

以下场景强制读取主库：

```text
刚刚修改密码
刚刚撤销 Session
刚刚禁用账号
刚刚修改角色
刚刚批准高权限
刚刚完成账号恢复
```

可以使用：

```text
ConsistencyContext
```

标记当前请求：

```text
PRIMARY_REQUIRED
REPLICA_ALLOWED
```

# 八、数据规模化设计

## 8.1 不要过早分库分表

优先优化顺序：

```text
正确索引
查询优化
分页
冷热分离
表分区
只读副本
归档
最后才是分片
```

身份系统存在大量全局唯一关系：

```text
邮箱
外部身份
Client ID
API Key Prefix
Credential ID
组织 Slug
```

过早按组织分片会显著增加全局唯一性、跨组织查询和账号登录复杂度。

---

## 8.2 优先分区的表

高增长表：

```text
identity_audit_event
identity_login_attempt
identity_security_event
identity_oauth_event
identity_provisioning_log
identity_outbox_event
identity_inbox_event
identity_plugin_execution
```

建议按：

```text
时间
+
可选 organization_id
```

分区。

例如：

```text
按月分区
按季度归档
```

核心用户和权限表不应因为审计数据增长而受到影响。

---

## 8.3 热数据与冷数据

热数据：

```text
用户状态
Session
Authenticator
Membership
Role
Permission
Token Family
当前 Grant
```

冷数据：

```text
历史审计
过期 Login Attempt
已完成 Provisioning Log
历史 Access Review
历史证据包
旧 OAuth Event
```

冷数据可以转移到：

```text
归档数据库
对象存储
审计仓库
SIEM
```

Identity 数据库只保留检索索引和引用。

---

## 8.4 分片触发条件

只有出现以下问题时才评估分片：

```text
单库容量达到明确上限
写吞吐无法通过纵向扩展解决
索引维护成为持续瓶颈
单组织成为极端热点
备份和恢复窗口不可接受
```

分片键优先考虑：

```text
organization_id
```

但以下全局索引需要独立处理：

```text
Email Directory
External Identity Directory
Client Directory
Credential Prefix Directory
```

# 九、Session 与共享状态

## 9.1 Session Store

支持两个实现：

```text
DatabaseSessionStore
RedisSessionStore
```

### DatabaseSessionStore

适用：

```text
社区版
小规模生产
Redis 不可用降级
```

### RedisSessionStore

适用：

```text
大量活跃 Session
多实例
低延迟 Session 查询
高频撤销
```

数据库仍保留 Session 的安全事实和审计。

Redis 可以作为：

```text
当前活动状态
短期索引
撤销缓存
```

不能成为唯一持久身份事实。

---

## 9.2 Redis 运行模式

小型高可用可以使用：

```text
Redis Primary + Replica + Sentinel
```

更大规模可以使用：

```text
Redis Cluster
```

Redis 官方文档说明基础复制默认是异步复制；Sentinel 提供监控和 Failover，Redis Cluster 提供分片和一定程度的故障可用性，但在多数主节点不可用等较大故障下仍可能整体不可用。

因此不能将 Redis 视为绝对不会丢失或永远可用的事实数据库。

---

## 9.3 Redis 故障降级

Redis 不可用时：

```text
新登录
    → 可回退数据库 Session Store

现有 Session
    → 根据数据库或本地短缓存验证

权限缓存
    → 回退数据库查询

限流
    → 回退节点级限流

Token 撤销
    → 查询数据库撤销状态
```

降级时：

```text
延迟可能升高
容量可能下降
但不能静默跳过安全校验
```

---

## 9.4 缓存内容

适合缓存：

```text
JWKS
用户安全版本
组织授权版本
成员有效权限
Client 配置
Scope 映射
Security Policy
Federation Metadata
```

不适合长期缓存：

```text
完整密码 Hash
TOTP Secret
Recovery Code
Client Secret
Signing Private Key
原始 SAML Assertion
```

---

## 9.5 缓存失效

缓存 Key 必须包含版本：

```text
permission:{userId}:{organizationId}:{authorizationVersion}
```

变化事件：

```text
membership.roles_changed
role.permissions_changed
organization.suspended
user.disabled
security_version.changed
```

发布缓存失效事件。

即使事件丢失，TTL 和版本检查也必须最终纠正。

# 十、可靠事件体系

P0 已有 Outbox。

P7 将其升级为：

```text
Database Transaction
    │
    ▼
Outbox
    │
    ▼
Event Relay
    │
    ▼
Kafka / Other Broker
    │
    ▼
Consumers
    │
    ▼
Inbox + Idempotent Handler
```

---

## 10.1 为什么保留 Outbox

即使接入 Kafka，也不要在业务事务中直接：

```text
更新数据库
然后发送 Kafka
```

中间可能出现：

```text
数据库成功，消息失败
消息成功，数据库回滚
```

正确模式：

```text
业务数据
+
Outbox Event
```

在同一数据库事务中提交。

---

## 10.2 Kafka 的定位

Kafka 适用于：

```text
高吞吐事件
多个消费者
重放
长时间保留
流式集成
```

Kafka 官方文档提供持久 Topic 和 Exactly-once 处理能力，但 Identity 对外部数据库、HTTP 和邮件等副作用仍应通过幂等键、Inbox 和业务唯一约束保证重复消费安全。

不要因为 Broker 声称支持 Exactly-once，就认为所有跨系统副作用天然只执行一次。

---

## 10.3 事件投递语义

Core Identity 采用：

```text
At-least-once Delivery
+
Idempotent Consumer
```

每个事件具有：

```text
event_id
event_type
event_version
aggregate_type
aggregate_id
organization_id
occurred_at
producer
trace_id
payload
```

---

## 10.4 Inbox

消费者记录已处理事件：

```text
consumer_name
event_id
processed_at
result
```

同一消费者重复收到相同 `event_id`：

```text
不重复执行副作用
返回已处理
```

---

## 10.5 死信与重放

失败事件进入：

```text
RETRYING
DEAD_LETTER
QUARANTINED
```

管理员可以：

```text
查看失败原因
修复配置
重新投递
跳过
标记已解决
```

重放必须保留原：

```text
event_id
trace_id
occurred_at
```

并增加：

```text
replay_id
replayed_by
replayed_at
```

---

# 十一、分布式任务

需要分布式协调的任务：

```text
过期 Grant 撤销
Refresh Token 清理
邀请过期
证书到期扫描
SCIM 重试
访问审查生成
证据采集
数据保留任务
Legal Hold 检查
插件健康检查
备份验证
```

---

## 11.1 任务执行模型

```text
Job Definition
    │
    ▼
Job Scheduler
    │
    ▼
Task Partition
    │
    ▼
Worker Node
    │
    ▼
Job Execution
```

任务状态必须入库。

不能只依赖：

```text
某个 Node 内存中的 @Scheduled
```

---

## 11.2 Leader Election

可支持：

```text
Database Lease
Kubernetes Lease
Broker Consumer Group
```

默认实现：

```text
Database Lease
```

企业 Kubernetes 部署可使用：

```text
Kubernetes Lease
```

Application 层只依赖：

```text
LeaderElectionPort
```

---

## 11.3 任务锁

任务 Lease 包含：

```text
owner_node_id
locked_until
heartbeat_at
fencing_token
```

Fencing Token 防止：

```text
旧节点暂停后恢复
继续以为自己仍然持有锁
```

每次重新获取 Lease：

```text
fencing_token + 1
```

下游写操作检查最新 Token。

---

## 11.4 任务幂等

任务必须能够安全重试。

例如过期 Grant：

```text
如果 Grant 已 EXPIRED
    → 返回成功，不重复撤销

如果 Session 已撤销
    → 返回成功

如果通知已经发送
    → 根据通知幂等键不重复发送
```

---

# 十二、容器和 Kubernetes 部署

Kubernetes 是 Enterprise HA 的一种参考部署方式，不是唯一方式。

## 12.1 Workload

建议：

```text
Identity Backend
    Deployment

Admin Backend
    Deployment

Web
    CDN / Static Hosting

Event Relay
    Deployment

Scheduled Worker
    Deployment

Migration
    Job
```

不要把数据库默认部署在同一个 Identity Pod 内。

---

## 12.2 探针

必须区分：

```text
Startup Probe
Liveness Probe
Readiness Probe
```

Kubernetes 使用这些探针判断容器是否已启动、是否需要重启以及是否应继续接收流量。

### Startup

检查：

```text
应用初始化完成
基础配置加载完成
必要密钥可用
```

### Readiness

检查：

```text
应用可以处理请求
数据库主连接可用
签名能力可用
关键 Schema 兼容
```

### Liveness

只检查：

```text
进程是否陷入不可恢复状态
```

不要因为某个非关键下游服务失败就让 Liveness 失败并反复重启。

---

## 12.3 优雅关闭

关闭流程：

```text
Readiness → false
    │
    ▼
停止接收新请求
    │
    ▼
等待进行中请求
    │
    ▼
停止消费新事件
    │
    ▼
提交或回滚当前事务
    │
    ▼
释放 Lease
    │
    ▼
关闭进程
```

Kubernetes 在容器终止时提供 `PreStop` 和终止宽限期；宽限期同时覆盖 PreStop 和应用正常退出，因此应用关闭时间必须纳入配置。

---

## 12.4 可用区分布

至少配置：

```text
Pod Anti-Affinity
Topology Spread Constraints
多个可用区
```

Topology Spread Constraints 可以控制 Pod 在节点、区域等故障域之间的分布。

---

## 12.5 PodDisruptionBudget

例如：

```text
minAvailable: 2
```

防止集群维护时同时驱逐过多 Identity 实例。

PodDisruptionBudget 用于限制自愿中断期间允许同时不可用的 Pod 数量。

---

## 12.6 自动扩缩容

HPA 可以根据：

```text
CPU
内存
HTTP 并发
请求延迟
事件积压
Token 签发速率
```

扩容。

Kubernetes HPA 通过调整 Deployment 或 StatefulSet 等工作负载的副本数匹配需求。

不要只按 CPU 扩缩 Identity。

例如：

```text
密码 Hash 验证
WebAuthn 验证
SAML XML 验证
Access Review 生成
```

有不同资源特征。

# 十四、Schema 零停机迁移

## 14.1 Expand and Contract

### Expand

```text
新增字段
新增表
新增索引
保持旧字段
新旧代码都能运行
```

### Migrate

```text
后台回填
双读或短期双写
校验数据
```

### Switch

```text
切换读取新字段
停止写旧字段
```

### Contract

```text
确认没有旧版本
删除旧字段
删除兼容代码
```

---

## 14.2 禁止单步破坏性迁移

不要在同一个版本：

```text
重命名字段
修改所有数据
删除旧字段
上线只认识新字段的代码
```

这会让 Rolling Update 中的新旧节点无法共存。

---

## 14.3 Schema Migration Runner

只能由一个执行者运行数据库迁移。

推荐：

```text
独立 Migration Job
```

而不是每个 Pod 启动时抢着执行高风险迁移。

普通 Pod 启动时只检查：

```text
Schema 是否处于兼容范围
```

---

## 14.4 大表迁移

大表增加索引或回填时：

```text
分批
限速
可暂停
可恢复
记录游标
监控数据库负载
```

迁移任务进入：

```text
identity_data_migration_job
```

不要在 Flyway 脚本里执行数小时不可恢复的数据回填。


# 十八、故障降级矩阵

## 18.1 Notification 不可用

```text
允许：
已有用户登录
Token 签发
权限判断

延迟：
验证邮件
密码重置邮件
安全通知
```

进入 Outbox 重试。

---

## 18.2 Billing 不可用

```text
允许：
已有账号登录
已有成员访问
已有 Entitlement 使用

限制：
新建收费组织
新增付费 SSO Connection
创建超额度 Service Account
```

不得因为 Billing 短暂故障让所有企业员工无法登录。

---

## 18.3 Workflow 不可用

```text
核心状态仍正常提交
工作流扩展动作进入队列
审批提醒延迟
复核任务延迟
```

Identity 自己拥有审批和 Grant 状态。

---

## 18.4 Storage 不可用

```text
头像和 Logo 使用默认图
证据导出暂缓
隐私导出暂缓
已有身份认证继续
```

---

## 18.5 Redis 不可用

```text
回退数据库
降低缓存
降低限流精度
发出 DEGRADED 状态
```

不能跳过 Session 或撤销检查。

---

## 18.6 Event Broker 不可用

```text
Outbox 继续写入
业务事务继续完成
Relay 暂停
队列恢复后重放
```

如果 Outbox 达到安全上限：

```text
拒绝部分非核心写操作
保留登录、撤销和安全处置
```

---

## 18.7 KMS 不可用

如果内存中已有有效签名 Key：

```text
在受控时间内继续签发短期 Token
禁止 Key Rotation
产生高危告警
```

如果没有可用签名 Key：

```text
不能签发新 Token
仍可验证已有 Token 的公钥签名
已有 Cookie Session 可按策略继续
```

绝不能临时生成未经治理的新生产密钥。

# 十九、数据库高可用

## 19.1 推荐拓扑

```text
Primary
├── Synchronous Standby，可选
├── Asynchronous Standby
└── Read Replica
```

PostgreSQL 文档支持一个或多个 Standby，并可通过同步 Standby 配置提高故障接管时的数据保障。

---

## 19.2 Failover

Failover 流程：

```text
检测 Primary 故障
    │
    ▼
确认不是网络分区误判
    │
    ▼
执行 Fencing
    │
    ▼
提升 Standby
    │
    ▼
更新数据库入口
    │
    ▼
应用重新连接
    │
    ▼
验证写入
    │
    ▼
记录 Failover Event
```

没有 Fencing 的自动 Failover 可能产生双主写入。

---

## 19.3 连接池

每个实例不能配置过大的连接池。

总连接数：

```text
实例数 × 每实例最大连接
```

必须小于数据库可承载范围。

扩容 Node 时同步调整：

```text
连接池
数据库连接上限
任务并发
```

# 二十、灾难恢复

## 20.1 RPO 与 RTO

必须为不同数据定义目标。

### 身份核心数据

```text
User
Credential
Authenticator
Membership
Role
Grant
Signing Key Metadata
```

要求最低 RPO。

### 审计和安全数据

```text
Audit
Security Event
Access Review
Evidence
```

需要长期保留，但恢复优先级可以与登录核心分开。

### 临时数据

```text
短期 Challenge
临时缓存
过期限流状态
```

可以容忍更高损失。

---

## 20.2 备份类型

```text
数据库逻辑备份
数据库物理备份
WAL / Binlog 连续归档
对象存储版本
密钥材料备份
配置备份
插件包备份
```

PostgreSQL 官方提供 SQL Dump、文件系统备份和连续归档/PITR 等不同备份方式；`pg_basebackup` 可以对运行中的数据库制作基础备份并用于恢复或 Standby。

---

## 20.3 备份加密

备份必须：

```text
传输加密
静态加密
独立访问控制
独立保留策略
校验和
备份清单
```

备份加密 Key 不能只保存在同一个数据库内。

---

## 20.4 恢复演练

定期执行：

```text
恢复到隔离环境
校验 Schema
校验用户数量
校验权限关系
校验 Signing Key
执行测试登录
执行 Token 签发
校验审计链
记录实际 RTO/RPO
```

没有恢复演练的备份不能被视为已经验证可用。

---

## 20.5 灾难模式

状态：

```text
NORMAL
DEGRADED
READ_ONLY
RECOVERY
DISASTER_FAILOVER
```

### READ_ONLY

允许：

```text
部分已有 Session 查询
审计查询
公开 JWKS
```

禁止：

```text
新用户注册
密码修改
权限修改
新 Token 长期授权
```

具体允许范围应根据故障类型决定。

# 二十一、多区域设计

## 21.1 P7 首选 Active-Passive

推荐：

```text
主区域：读写
灾备区域：待命
```

而不是一开始实现：

```text
多区域 Active-Active 写入
```

身份系统存在：

```text
邮箱唯一性
Refresh Token Rotation
Session 撤销
授权版本
账号恢复
一次性 Token
```

这些状态对一致性要求很高。

---

## 21.2 多区域读优化

可以：

```text
JWKS 多区域缓存
静态前端多区域
公开 Metadata 多区域
只读报表副本
```

核心写入仍进入主区域。

---

## 21.3 区域切换

切换前：

```text
确认主区域不可恢复
Fencing 主区域
确认复制位置
提升灾备数据库
激活签名服务
更新 DNS 或全局入口
验证 Token
验证登录
```

恢复主区域后不能直接重新加入写入。

需要重新同步并作为 Standby 加入。

# 二十三、扩展平台总体原则

P7 开放扩展，但不能允许插件直接成为 Identity 内核的一部分。

扩展层级：

```text
Level 1：配置扩展
Level 2：声明式扩展
Level 3：受信任进程内扩展
Level 4：隔离进程外扩展
```

---

## 23.1 Level 1：配置扩展

例如：

```text
密码策略参数
通知模板
Risk Rule
Claim Mapping
Access Package 模板
```

不执行第三方代码。

风险最低。

---

## 23.2 Level 2：声明式扩展

例如：

```text
Webhook
事件订阅
权限清单
UI Schema
字段映射
工作流触发
```

通过受控声明执行。

---

## 23.3 Level 3：受信任进程内扩展

适合官方或企业自有可信扩展：

```text
IdentityProviderAdapter
RiskSignalProvider
AuditSink
PasswordPolicyProvider
AuthenticatorMetadataProvider
NotificationAdapter
```

Java 的 `ServiceLoader` 可以根据稳定服务接口发现和加载 Provider。

仅允许：

```text
官方签名插件
企业管理员明确安装
完整重启后加载
严格版本匹配
```

---

## 23.4 Level 4：进程外扩展

适合不完全可信的第三方扩展：

```text
Webhook Service
gRPC Extension
HTTP Extension
独立容器
独立进程
```

通过：

```text
网络隔离
Service Account
Scope
超时
限流
资源配额
```

控制。

---

## 23.5 不在同 JVM 中运行不可信插件

OpenJDK 已将 Security Manager 弃用并推进永久禁用，官方建议此类不可信代码使用容器、虚拟机等进程外隔离机制，而不是依赖旧式 JVM 沙箱。

因此：

```text
第三方插件
未知来源插件
Marketplace 社区插件
```

不得直接拥有 Identity JVM 的完整 Classpath 和进程权限。

# 六十、P7 最重要的注意点

## 1. 高可用不等于部署两个 Pod

如果两个 Pod：

```text
共享同一个单点数据库
共享同一个本地磁盘
Session 只在内存
任务会重复运行
```

那么它并不是真正高可用。

---

## 2. Redis 不能成为身份事实来源

Redis 是：

```text
缓存
Session 加速
撤销传播
限流
```

不是：

```text
用户
权限
Grant
认证器
审计
```

的唯一存储。

---

## 3. Kafka 不能消除幂等设计

Broker 可以提高可靠性，但消费者仍必须处理：

```text
重复
乱序
延迟
重放
```

---

## 4. 不要过早做多区域 Active-Active

账号恢复、Refresh Token Rotation 和一次性 Token 对一致性要求极高。

优先 Active-Passive。

---

## 5. 插件不是普通依赖包

插件具有：

```text
代码执行
数据访问
网络访问
身份权限
升级风险
供应链风险
```

必须像独立应用一样治理。

---

## 6. 不要依赖 Java Security Manager 隔离不可信插件

不可信插件必须进程外隔离。

---

## 7. 运维控制台不能取代专业基础设施平台

Identity Admin Web 可以展示和编排有限操作。

不能替代：

```text
数据库管理平台
Kubernetes 管理平台
日志平台
SIEM
Secret Manager
```

---

## 8. 灾备的完成标志是成功恢复

不是：

```text
备份任务显示绿色
```

而是：

```text
可以恢复
可以登录
可以签发 Token
权限数据正确
审计链可验证
```

---
