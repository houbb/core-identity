# Core Identity P5：企业 SSO 与身份联合

版本：P5
前置依赖：P0 工程边界、P1 Identity MVP、P2 组织权限、P3 平台身份服务、P4 账号安全
目标：支持企业通过 OIDC、SAML 和 SCIM 接入现有身份系统，实现统一登录、自动建号、成员同步、离职停权和身份生命周期管理。

---

# 一、P5 核心目标

P4 已经解决：

```text
用户如何安全证明自己
认证强度是否足够
当前登录是否可信
账号被接管后如何恢复
```

P5 继续解决：

```text
企业能否使用自己的身份提供商登录？
员工是否还需要单独维护 Core Identity 密码？
用户属于哪个企业组织？
企业 IdP 的用户如何映射到本地 User？
首次登录是否自动创建成员？
员工离职后如何立即停止访问？
企业组如何映射为 Core Identity 角色？
多个身份提供商如何路由？
IdP 不可用时如何应急？
```

P5 必须完成：

```text
企业域名验证
身份提供商连接
OIDC 企业 SSO
SAML 2.0 企业 SSO
身份发现与登录路由
外部身份绑定
账号冲突处理
JIT Provisioning
SSO 强制策略
SCIM 2.0 User Provisioning
SCIM 2.0 Group Provisioning
外部组与本地角色映射
离职与停权
IdP 证书和密钥轮换
单点退出基础
企业 SSO 诊断
企业身份审计
```

OpenID Connect 在 OAuth 2.0 之上提供身份层，通过 ID Token 和 Claims 表达已认证用户；SAML 2.0 则定义安全断言及其交换框架；SCIM 使用标准 HTTP 协议管理跨域用户和组资源。

# 二、P5 暂时不做什么

P5 不实现：

```text
完整身份治理审批
定期访问审查
职责分离
临时权限审批
复杂 HRIS 集成
多级组织树同步
身份数据湖
跨企业 B2B 联邦网络
政府级身份保证认证
自定义身份协议
自研 SAML/XML 加密库
```

这些属于 P6、P7 或独立企业集成阶段。

P5 的范围是：

> Core Identity 作为企业 IdP 的 Relying Party / Service Provider，同时继续作为下游 Core 服务的统一身份发行者。

# 三、最重要的身份代理架构

企业登录链路：

```text
企业员工
    │
    ▼
Core Identity 登录入口
    │
    ▼
识别所属组织和 IdP
    │
    ▼
企业 OIDC / SAML IdP
    │
    ▼
企业 IdP 完成认证
    │
    ▼
Core Identity 验证身份断言
    │
    ▼
映射 External Identity
    │
    ▼
映射或创建本地 User
    │
    ▼
映射 Organization / Membership / Role
    │
    ▼
创建 Core Identity Session
    │
    ▼
由 Core Identity 向其他 Core 签发 Token
```

必须坚持：

> 上游 IdP Token 不直接传给 Billing、Storage、AI Gateway 等下游服务。

下游服务只信任：

```text
Core Identity Session
Core Identity Access Token
Core Identity API Key
Core Identity Service Account
```

这样可以确保：

```text
统一 issuer
统一 subject
统一 organization_id
统一 permission
统一 scope
统一撤销机制
统一审计
```

# 四、Core Identity 在 P5 中的双重角色

P3 以后，Core Identity 已经是：

```text
OIDC Provider
OAuth Authorization Server
```

它向下游应用签发身份。

P5 又增加：

```text
OIDC Relying Party
SAML Service Provider
SCIM Service Provider
```

它从企业身份系统接收身份。

完整结构：

```text
企业 IdP
    │
    │ OIDC / SAML
    ▼
Core Identity
    │
    │ OAuth / OIDC
    ▼
Core Platform Apps
```

这是 Identity Broker 模式。

# 七、Federation Connection 模型

一个企业组织可以配置一个或多个身份提供商连接。

例如：

```text
Acme Corporation
├── Microsoft Entra ID - OIDC
├── Okta - SAML
└── Contractor IdP - OIDC
```

连接类型：

```text
OIDC
SAML
```

后续可以扩展：

```text
LDAP Gateway
CAS
Custom Enterprise Adapter
```

但 P5 不实现。

---

## 7.1 Connection 状态

```text
DRAFT
VALIDATING
ACTIVE
DEGRADED
SUSPENDED
ERROR
DELETED
```

### DRAFT

配置尚未完成，不能用于登录。

### VALIDATING

正在测试 Discovery、Metadata、证书或回调。

### ACTIVE

可以用于企业登录。

### DEGRADED

仍可使用，但存在：

```text
证书即将过期
Discovery 暂时不可访问
部分 Claim 缺失
Logout 失败
SCIM 同步异常
```

### SUSPENDED

组织管理员或平台管理员主动停用。

### ERROR

配置无法用于认证。

---

## 7.2 Connection 所有权

每个 Connection 必须绑定：

```text
organization_id
```

平台级 Connection 可用于：

```text
平台员工
内部管理团队
共享企业 IdP 模板
```

普通组织管理员不能查看或修改其他组织 Connection。

# 八、企业域名验证

域名用于：

```text
证明企业对域名的控制权
发现用户可能所属的组织
防止其他组织冒用企业域名
支持强制 SSO
支持 JIT 邮箱限制
```

但必须明确：

> 域名所有权不能单独证明某个具体用户身份，也不能单独作为已有账号自动绑定依据。

---

## 8.1 域名验证方式

P5 推荐优先支持：

```text
DNS TXT
```

例如：

```text
_core-identity-verification.example.com
```

值：

```text
core-identity-verification=随机挑战值
```

可选备用：

```text
企业邮箱管理员确认
指定文件验证
```

生产企业级强制 SSO建议以 DNS 验证为准。

---

## 8.2 域名状态

```text
PENDING
VERIFIED
FAILED
EXPIRED
REVOKED
CONFLICT
```

---

## 8.3 域名唯一性

一个主域名默认只能被一个 ACTIVE 组织声明。

例如：

```text
example.com
```

不能同时归属两个无关联组织。

子域名可以独立配置：

```text
partners.example.com
employees.example.com
```

是否允许父子域名分别归属，需要平台策略明确。

---

## 8.4 域名验证流程

```text
组织管理员输入域名
    │
    ▼
Identity 生成随机挑战
    │
    ▼
管理员添加 DNS TXT
    │
    ▼
点击验证
    │
    ▼
Identity 查询 DNS
    │
    ├── 匹配
    │     → VERIFIED
    │
    └── 不匹配
          → 保持 PENDING
```

验证后定期复查。

不要因为一次验证成功就永久认为域名归属不变。

# 九、身份发现与登录路由

## 9.1 入口形式

支持三种入口。

### 通用登录入口

```text
/login
```

用户输入邮箱：

```text
ayse@example.com
```

系统识别：

```text
example.com → Acme SSO
```

然后跳转企业 IdP。

### 组织专属入口

```text
/sso/acme
```

直接进入指定组织 SSO。

### IdP 专属入口

```text
/sso/connections/{connectionKey}
```

适合一个组织拥有多个 IdP。

---

## 9.2 Home Realm Discovery

发现过程：

```text
输入邮箱或组织标识
    │
    ▼
标准化域名
    │
    ▼
查找 VERIFIED Domain
    │
    ▼
查找 ACTIVE Federation Connection
    │
    ├── 一个
    │     → 自动跳转
    │
    ├── 多个
    │     → 展示身份提供商选择
    │
    └── 没有
          → 使用本地登录
```

---

## 9.3 防止邮箱枚举

通用登录页不能公开确认：

```text
某邮箱是否存在
某邮箱是否已经是组织成员
```

但可以根据已验证的公开企业域名路由到 SSO。

界面提示：

```text
该组织使用企业登录。
```

而不是：

```text
该邮箱是平台用户，ID 为……
```

---

## 9.4 多组织冲突

同一个邮箱可能：

```text
属于多个组织
多个组织配置了不同 IdP
使用个人账号和企业账号
```

处理原则：

```text
组织专属链接优先
已知 Session 组织优先
用户明确选择优先
不自动猜测高风险组织
```

# 十、OIDC 企业 SSO

OIDC Connection 需要：

```text
Issuer
Discovery URI
Client ID
Client Secret
Redirect URI
Requested Scopes
Claim Mapping
Logout Configuration
```

OIDC Discovery 提供 Provider 端点和能力元数据，Core Identity 应校验发现文档中的 issuer 与预期 issuer 一致。

---

## 10.1 OIDC 登录流程

```text
用户选择企业登录
    │
    ▼
Core Identity 创建 Federation State
    │
    ▼
生成 state、nonce、PKCE
    │
    ▼
重定向企业 Authorization Endpoint
    │
    ▼
企业 IdP 完成认证
    │
    ▼
回调 Core Identity
    │
    ▼
验证 state
    │
    ▼
使用 Authorization Code 换 Token
    │
    ▼
验证 ID Token
    │
    ▼
映射 External Identity
    │
    ▼
创建 Core Identity Session
```

即使 Core Identity 是服务端 Confidential Client，也建议保留 PKCE，作为额外防护；OAuth 2.0 的当前安全最佳实践由 RFC 9700 给出。

---

## 10.2 必须验证的 OIDC 内容

```text
issuer
signature
kid
audience
azp
expiration
issued_at
nonce
state
authorization code
redirect URI
```

可选验证：

```text
acr
amr
auth_time
email_verified
```

不能仅调用 UserInfo Endpoint 后相信返回的邮箱。

---

## 10.3 外部身份唯一键

OIDC External Identity 的唯一键必须是：

```text
issuer + subject
```

即：

```text
iss + sub
```

不能使用邮箱作为唯一外部身份键。

邮箱可能：

```text
改变
被重新分配
大小写变化
使用别名
跨租户重复
```

---

## 10.4 上游 Token 处理

登录完成后，默认只需要：

```text
ID Token 验证结果
必要的 UserInfo Claims
```

如果不需要调用上游 API：

```text
不长期保存 Upstream Access Token
不长期保存 Upstream Refresh Token
```

确实需要 Back-channel Logout 或上游 API 时：

```text
必须加密保存
最小 Scope
明确过期
支持撤销
```

---

## 10.5 OIDC Authentication Context

如果上游 IdP 返回：

```text
acr
amr
auth_time
```

可以映射为 P4 的：

```text
authentication_level
authentication_methods
strong_auth_at
```

但不能盲目信任任意字符串。

每个 Connection 需要配置：

```text
上游 acr → Core AUTH_LEVEL 映射
```

例如：

```text
urn:example:mfa
→ AUTH_LEVEL_2
```

没有明确映射时，采用保守等级。

# 十一、SAML 2.0 企业 SSO

SAML Connection 需要：

```text
SP Entity ID
ACS URL
SP Metadata
IdP Entity ID
IdP SSO URL
IdP Certificate
NameID Policy
Attribute Mapping
Signing Policy
Encryption Policy
Logout Configuration
```

SAML 定义主体断言及协议、绑定、配置文件和元数据；Spring Security 当前支持应用作为 SAML 2.0 Relying Party，并提供登录、Metadata 与 Logout 能力。

---

## 11.1 首期支持范围

P5 优先支持：

```text
SP-Initiated SSO
HTTP-Redirect AuthnRequest
HTTP-POST Response
Signed Response
Signed Assertion
Metadata Import
Metadata Export
```

随后支持：

```text
IdP-Initiated SSO
Encrypted Assertion
Single Logout
```

IdP-Initiated SSO 缺少由本平台发起并关联的完整请求上下文，必须采用更严格的 RelayState 和重放防护。

---

## 11.2 SAML 登录流程

```text
用户选择企业 SSO
    │
    ▼
Core Identity 创建 AuthnRequest
    │
    ▼
记录 Request ID 与 RelayState
    │
    ▼
重定向 IdP
    │
    ▼
IdP 返回 SAML Response
    │
    ▼
Core Identity 验证：
    XML Signature
    Certificate
    InResponseTo
    Destination
    Recipient
    Audience
    Issuer
    Conditions
    NotBefore / NotOnOrAfter
    SubjectConfirmation
    Replay
    │
    ▼
映射 NameID 和 Attributes
    │
    ▼
绑定 External Identity
    │
    ▼
创建 Core Identity Session
```

---

## 11.3 SAML 必须防护

```text
签名绕过
XML Signature Wrapping
Assertion 重放
Response 重放
错误 Audience
错误 Recipient
错误 Destination
伪造 Issuer
过大时间漂移
不受信任证书
弱签名算法
恶意 XML 实体
```

禁止：

```text
手写 XML Signature 验证
只验证外层 Response 而忽略 Assertion
只读取邮箱 Attribute 就登录
跳过 Audience 和 Recipient 验证
```

实现应使用 Spring Security SAML2 与成熟 OpenSAML 能力，不自研 SAML 协议解析。Spring Security 官方文档明确将其 SAML 登录能力定位为 Relying Party 支持。

---

## 11.4 NameID 与外部身份

SAML External Identity 唯一键优先使用：

```text
IdP Entity ID + Persistent NameID
```

如果企业只能提供：

```text
email
transient NameID
employeeId
```

必须让管理员明确配置唯一标识 Claim。

推荐优先级：

```text
稳定 employee/object ID
Persistent NameID
不可变自定义 ID
邮箱，仅作为最后方案
```


必须通过：

```text

至少一次测试登录成功
Break-glass 账号存在
```

不能刚保存一个 Issuer 就立即对全组织强制 SSO。

# 十三、External Identity 模型

一个本地 User 可以绑定多个外部身份。

例如：

```text
User Ayşe
├── Local Password
├── Microsoft Entra ID / tenant-a / subject-123
├── Okta / organization-b / subject-456
└── Google / subject-789
```

External Identity 表示：

```text
哪个 IdP
哪个 Connection
哪个外部 Subject
绑定哪个本地 User
```

---

## 13.1 External Identity 状态

```text
ACTIVE
SUSPENDED
UNLINKED
CONFLICT
ORPHANED
```

### ACTIVE

正常可用于登录。

### SUSPENDED

IdP 或管理员暂停。

### UNLINKED

已经解除绑定。

### CONFLICT

同一外部身份或邮箱发生冲突，需要人工处理。

### ORPHANED

Connection 被删除，但历史身份仍保留。

# 十四、账号匹配与绑定规则

这是 P5 最危险的部分。

错误设计：

```text
只要外部 IdP 返回的 email
和本地 User.email 相同
就自动绑定
```

这可能导致账号接管。

---

## 14.1 安全绑定优先级

### 第一优先级

已有匹配：

```text
connection_id + external_subject
```

直接登录对应 User。

### 第二优先级

当前用户已经登录，并主动执行：

```text
绑定企业身份
```

完成强再认证后绑定。

### 第三优先级

经过验证的企业 Connection，使用：

```text
已验证域名
email_verified
管理员启用可信自动链接
```

仍建议要求用户确认或管理员预先关联。

### 默认情况

发现同邮箱本地账号时：

```text
不自动绑定
进入账号关联确认
```

---

## 14.2 账号关联确认

用户必须证明：

```text
能够登录现有本地账号
或完成现有强认证器验证
```

然后确认：

```text
将 Acme 企业身份绑定到当前账户
```

如果无法证明现有账号：

```text
进入管理员审核
不能直接创建第二个同邮箱账号
```

---

## 14.3 禁止通过 IdP Claim 提升平台身份

上游 IdP 返回：

```text
isAdmin=true
role=superuser
```

不能直接映射为：

```text
Platform SUPER_ADMIN
```

平台管理员身份只能通过平台受控流程授予。

企业 Claim 最多映射为：

```text
该企业组织内部角色
```

## 15.1 属性来源策略

每个属性指定所有权：


只由用户或本地管理员维护。

### SCIM

### EXTERNAL_ALWAYS

右侧：

提供：


# 十六、JIT Provisioning

JIT 表示：

> 用户第一次通过企业 SSO 成功登录时，自动创建或加入组织。

---

## 16.1 JIT 流程

```text
外部认证成功
    │
    ▼
查找 External Identity
    │
    ├── 已存在
    │     → 更新允许的属性
    │
    └── 不存在
          ▼
       检查 JIT Policy
          │
          ├── 禁止
          │     → 拒绝并提示联系管理员
          │
          └── 允许
                ▼
             检查域名
                ▼
             检查账号冲突
                ▼
             创建或绑定 User
                ▼
             创建 Membership
                ▼
             分配默认角色
                ▼
             创建 External Identity
```

---

## 16.2 JIT 策略

```text
ENABLED
DISABLED
REQUIRE_APPROVAL
```

条件：

```text
允许的邮箱域名
允许的 IdP Tenant
默认组织
默认角色
是否允许创建新 User
是否允许链接已有 User
是否要求 email_verified
是否同步显示名称
是否同步外部组
```

---

## 16.3 默认角色

JIT 默认角色应是：

```text
MEMBER
或 VIEWER
```

不能默认：

```text
ADMIN
OWNER
```

OWNER 永远不能通过 JIT 自动授予。

---

## 16.4 JIT 与 Billing 席位

如果企业套餐限制成员数量：

```text
JIT 登录前
→ 检查 Billing Entitlement / Seat
```

但要定义 Billing 不可用时的行为。

推荐：

```text
已有成员继续登录
新 JIT 成员暂缓创建
显示明确错误
```

不要无上限创建成员，也不要让 Billing 故障锁死已有用户。

# 十七、SSO 强制策略

组织可以配置：

```text
OPTIONAL
REQUIRED_FOR_MEMBERS
REQUIRED_FOR_DOMAINS
REQUIRED_FOR_ALL_EXCEPT_BREAK_GLASS
```

---

## 17.1 OPTIONAL

用户可以：

```text
企业 SSO
本地密码
Passkey
```

---

## 17.2 REQUIRED_FOR_MEMBERS

该组织成员访问组织资源时必须通过指定 IdP。

用户仍可使用本地方式访问：

```text
个人空间
其他组织
账户安全设置
```

---

## 17.3 REQUIRED_FOR_DOMAINS

使用已验证企业域名的账号必须通过企业 SSO。

此策略风险较高，需要明确处理个人用户和历史账号。

---

## 17.4 Break-glass 账号

每个强制 SSO 组织至少保留：

```text
一个或两个紧急本地管理员账号
```

要求：

```text
强 MFA
Passkey 或硬件安全密钥
不受企业 SSO 强制策略影响
不得用于日常登录
每次使用产生 CRITICAL 安全事件
定期验证可用性
```

不要在 IdP 故障时自动允许所有用户回退密码登录。

否则强制 SSO 可以被轻易绕过。

---

## 17.5 策略发布流程

```text
配置 SSO
    │
    ▼
完成测试
    │
    ▼
确认 Break-glass
    │
    ▼
预览影响成员
    │
    ▼
设置宽限期
    │
    ▼
通知成员
    │
    ▼
正式强制
```

# 十九、SCIM 2.0 定位

OIDC 和 SAML 解决：

```text
用户登录时是谁
```

SCIM 解决：

```text
登录之前和离职之后，用户和组应该处于什么状态
```

SCIM Core Schema 定义 User、Group 等资源结构；SCIM Protocol 定义基于 HTTP 的创建、查询、修改和删除语义，并支持过滤、分页与 PATCH 等能力。RFC 9865 又为 SCIM 增加了可选的游标分页机制。

---

# 二十、SCIM Service Provider

Core Identity 提供：

```text
/scim/v2/ServiceProviderConfig
/scim/v2/ResourceTypes
/scim/v2/Schemas

/scim/v2/Users
/scim/v2/Groups
```

企业 IdP 作为 SCIM Client。

Core Identity 作为 SCIM Service Provider。

---

## 20.1 P5 支持的 SCIM User 操作

```text
POST   /Users
GET    /Users/{id}
GET    /Users
PUT    /Users/{id}
PATCH  /Users/{id}
DELETE /Users/{id}
```

`DELETE` 默认不物理删除本地 User。

处理为：

```text
Membership 停用
External Identity 停用
Session 撤销
Refresh Token 撤销
SCIM Resource 标记停用
```

---

## 20.2 SCIM User 关键字段

```text
id
externalId
userName
active
name
displayName
emails
locale
timezone
groups
meta
```

本地映射：

```text
SCIM id
→ SCIM Resource ID

externalId
→ 企业 IdP 侧不可变 ID

userName
→ 企业登录名

active
→ 企业成员是否有效
```

---

## 20.3 SCIM 唯一性

同一 SCIM Connection 内：

```text
externalId
```

必须唯一。

跨 Connection 可以重复。

推荐唯一约束：

```text
connection_id + resource_type + external_id
```

---

## 20.4 SCIM 创建 User

流程：

```text
SCIM POST /Users
    │
    ▼
验证 SCIM Token
    │
    ▼
验证 Organization
    │
    ▼
检查 externalId
    │
    ▼
检查 email / userName 冲突
    │
    ▼
创建或关联 User
    │
    ▼
创建 Membership
    │
    ▼
创建 External Identity 或 Provisioned Identity
    │
    ▼
应用默认角色
    │
    ▼
返回 SCIM Resource
```

---

## 20.5 SCIM 停用 User

企业 IdP 发送：

```json
{
  "active": false
}
```

Core Identity 执行：

```text
Membership → SUSPENDED
External Identity → SUSPENDED
撤销该组织上下文 Session
撤销相关 Refresh Token
暂停组织级 API Key
暂停组织级 Service Credential，若属于该用户
发布 member.deprovisioned
```

如果用户还属于其他组织：

```text
不禁用整个 User
不删除个人空间
不影响其他组织成员身份
```

这是极其重要的边界。

---

## 20.6 SCIM 删除与全局账号删除

企业从组织中删除员工，不等于：

```text
删除整个 Core Identity User
```

SCIM 只管理：

```text
该企业组织下的 Membership 和外部身份
```

全局 User 删除必须由独立隐私与治理流程处理。

# 二十一、SCIM Group Provisioning

支持：

```text
创建 Group
更新 Group
删除 Group
同步 Group Members
```

企业 Group 不直接等于 Core Identity Role。

正确模型：

```text
External Group
    │
    ▼
Group-Role Mapping
    │
    ▼
Core Identity Role
```

例如：

```text
Entra Group: Finance-Managers
    ↓
Core Role: Billing Manager
```

---

## 21.1 为什么不直接将 Group 当作 Role

企业组可能用于：

```text
邮件列表
部门
项目
地理位置
许可证
安全组
```

并不都代表平台权限。

通过 Mapping 可以明确：

```text
哪些外部组影响平台权限
映射到什么角色
映射范围是什么
```

---

## 21.2 Group Mapping 策略

```text
ADD_ONLY
AUTHORITATIVE
```

### ADD_ONLY

外部组只增加角色，不移除本地手工角色。

### AUTHORITATIVE

外部组决定指定角色集合。

用户离开外部组后，映射角色被移除。

推荐默认：

```text
ADD_ONLY
```

企业明确选择后才使用：

```text
AUTHORITATIVE
```

---

## 21.3 角色保护

外部组不能授予：

```text
OWNER
Platform SUPER_ADMIN
Break-glass Admin
```

是否允许授予组织 ADMIN：

```text
默认禁止
由平台策略控制
```

---

# 二十四、JIT 与 SCIM 的优先级

企业同时启用 JIT 和 SCIM 时，必须定义权威顺序。

```text

→ 身份生命周期和组织成员状态的权威来源

→ 登录时补充创建和更新的兜底机制

Local
→ 用户偏好和非企业托管属性
```

## 24.1 已经 SCIM 管理的用户

SSO 登录时：

```text

不能自行恢复已被 SCIM 停用的 Membership
```

```text

```

# 六十二、事务设计

## 62.1 OIDC / SAML 首次登录

同一事务：

```text
锁定 Federation State
验证外部身份结果
查找 External Identity
检查账号冲突
创建或关联 User
创建或恢复 Membership
创建 External Identity
应用 JIT 默认角色
更新允许同步的属性
创建 Core Session
创建 Federated Session
写 Audit
写 Security Event
写 Outbox
```

---

## 62.2 Existing External Identity 登录

同一事务：

```text
锁定 External Identity
检查状态
检查 Membership
检查 SSO Policy
更新外部属性
更新最近登录
创建 Core Session
创建 Federated Session
写登录审计
```

SCIM 已停用 Membership 时必须拒绝，不能自动恢复。

---

## 62.3 SCIM User 创建

同一事务：

```text
验证 SCIM Client
验证 externalId
检查账号冲突
创建或关联 User
创建 Membership
创建 SCIM Resource
```

# 七十二、安全注意点

## 72.1 不要用邮箱作为外部身份主键

OIDC 使用：

```text
issuer + subject
```

SAML 使用：

```text
IdP Entity ID + Persistent NameID / 不可变 ID
```

邮箱只用于展示和辅助匹配。

---

## 72.2 不要自动绑定所有同邮箱账号

域名验证和 email_verified 都不足以覆盖所有账号接管场景。

已有账号默认要求：

```text
现有账号认证
强再认证
或管理员审核
```

---

## 72.3 不要信任上游角色名称

企业 IdP 的：

```text
admin
superuser
owner
```

不能直接映射为平台管理员或组织所有者。

---

## 72.4 不要将 IdP Token 传给业务 Core

必须换成 Core Identity 自己的 Session 和 Token。

---

## 72.5 不要在 IdP 故障时静默回退密码

对于强制 SSO 组织，这会形成身份策略绕过。

只允许经过严格保护的 Break-glass 账号。

---

## 72.6 不要让 JIT 恢复 SCIM 已停用成员

SCIM 生命周期优先级高于 JIT 登录创建。

---

## 72.7 不要手写 SAML 安全验证

使用成熟协议实现，并完整验证签名、Audience、Recipient、时间和重放。

# 七十六、P5 验收标准

## 企业 SSO

```text
组织可以验证企业域名
组织可以配置 OIDC IdP
组织可以配置 SAML IdP
用户可以通过企业 IdP 登录
External Identity 可以稳定映射本地 User
下游 Core 只接收 Core Identity Token
```

## JIT

```text
首次 SSO 可以创建用户和成员
JIT 默认角色受到限制
已有账号不会被不安全自动绑定
已停用 SCIM 成员不会被 JIT 恢复
```

## SCIM

```text
企业可以创建、更新、停用用户
企业可以同步 Group
Group 可以映射本地 Role
SCIM 停用可以撤销组织访问
不会误删用户的其他组织身份
```

## 强制 SSO

```text
组织可以强制成员使用企业 SSO
策略支持宽限期
IdP 故障不会自动降级为弱登录
组织拥有安全的 Break-glass 账号
Break-glass 使用会产生高危事件
```

## 运维

```text
OIDC 和 SAML 都有测试工具
证书和 Secret 支持轮换
证书到期可以提前提醒
SCIM 错误可以查询和重试
平台管理员可以治理风险 Connection
```

## 边界

```text
Admin Backend 不访问 Identity 数据库
外部 IdP 不直接决定本地平台管理员
外部 Token 不传给其他 Core
Identity Backend 是唯一身份映射中心
```

# 七十七、P5 最重要的注意点

## 1. SSO 与 SCIM 是两件事

```text
SSO
证明当前登录者是谁

SCIM
管理用户是否应该存在和拥有访问权
```

只接 SSO 没有 SCIM，员工离职后仍可能保留平台成员身份。

---

## 2. Federation 不等于放弃本地身份模型

上游 IdP 负责认证。

Core Identity 仍负责：

```text
User ID
Organization
Membership
Role
Permission
Session
Token
Audit
```

---

## 3. 邮箱不是可靠的永恒身份 ID

邮箱可以用于发现和辅助匹配，但不能作为外部身份唯一主键。

---

## 4. 不要过早强制 SSO

必须先完成：

```text
域名验证
登录测试
账号冲突检查
Break-glass
影响预览
成员通知
```

---

## 5. SCIM 停用不能删除全局 User

它只能终止对应企业组织中的访问。

---

## 6. 企业组不能直接变成平台超级权限

Group 必须通过受控映射进入本地 Role，并保护 OWNER、平台管理员等关键身份。

---

## 7. 外部认证强度必须保守映射

不能因为 IdP 返回字符串：

```text
mfa
```

就默认认为满足最高认证等级。

---

# 七十八、P5 最终成果

P5 完成后：

```text
core-identity-backend

成为企业身份代理、OIDC Relying Party、
SAML Service Provider、SCIM Service Provider
和外部身份生命周期事实中心。
```

```text
core-identity-web

成为组织管理员配置企业 SSO、SCIM、

```

```text
core-identity-admin-backend

成为跨组织 Federation、证书、同步异常、
身份冲突与企业身份事故的治理编排层。
```

```text
core-identity-admin-web

成为平台企业身份连接、Provisioning、
证书风险和域名冲突的运营控制台。
```

P5 最终形成三条企业身份链路：

```text
企业 IdP
→ OIDC / SAML
→ External Identity
→ Local User
→ Core Session
```

```text
企业目录
→ SCIM User / Group
→ Membership / Role
→ Permission
```

```text

→ SCIM active=false
→ Membership 停用
→ Session / Token 撤销
→ 组织访问终止
```

到这一阶段，Core Identity 才真正从平台身份服务升级为：

> 能够接住企业既有身份体系，同时仍然守住 Core Platform 自己的用户、组织、权限和安全边界。
