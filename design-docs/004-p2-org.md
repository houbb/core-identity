# Core Identity P2：组织与权限体系详细设计

版本：P2
前置依赖：P0 工程边界、P1 Identity MVP
目标：将个人身份系统升级为支持多组织、成员协作和组织级 RBAC 的统一权限中心。

---

# 一、P2 核心目标

P1 已经完成：

```text
用户
邮箱
凭证
会话
个人空间
平台管理员
```

P2 在此基础上增加：

```text
团队组织
多组织成员关系
组织切换
组织邀请
组织所有权
成员管理
角色管理
权限目录
角色权限
成员角色
组织级授权判断
跨 Core 权限同步
```

P2 完成后，平台必须能够稳定回答：

```text
这个用户属于哪些组织？
当前请求代表哪个组织？
用户在该组织中是什么成员状态？
用户拥有哪些角色？
这些角色授予哪些权限？
当前操作是否允许？
```

# 三、P2 的授权模型

P2 使用：

```text
User
  ↓
Membership
  ↓
Role
  ↓
Permission
```

完整关系：

```text
User
  │
  │ belongs through
  ▼
Membership
  │
  │ assigned
  ▼
Role
  │
  │ contains
  ▼
Permission
```

同时，每个请求必须具有：

```text
Current User
Current Organization
Required Permission
```

授权判断公式：

```text
Allow =
    User 有效
    AND Organization 有效
    AND Membership 有效
    AND Membership 拥有的任一 Role
        包含 Required Permission
```

P2 不支持：

```text
显式拒绝优先
复杂条件表达式
按字段授权
按资源实例授权
基于时间授权
基于 IP 授权
```

这些暂时不要混入基础 RBAC。

# 五、组织模型

## 5.1 组织类型

P2 支持：

```text
PERSONAL
TEAM
```

### PERSONAL

P1 自动创建的个人空间。

特点：

```text
每个用户最多一个
不允许邀请成员
不允许转移所有权
不允许主动解散
不显示复杂组织管理
```

### TEAM

P2 新增的团队组织。

特点：

```text
可以邀请成员
可以创建自定义角色
可以调整成员角色
可以转移所有权
可以解散
可以被平台冻结
```

---

## 5.2 组织状态

```text
ACTIVE
SUSPENDED
PENDING_DELETION
DELETED
```

### ACTIVE

组织正常使用。

### SUSPENDED

由平台管理员冻结。

效果：

```text
成员不能执行组织写操作
组织资源不能新增
原有数据保持
平台管理员仍可查询
```

是否允许只读访问由各 Core 统一遵循。

P2 建议：

```text
默认允许组织成员读取已有数据
禁止新增、修改和删除
```

涉及安全风险时可以配置为全部拒绝。

### PENDING_DELETION

组织已申请解散，处于冷静期。

### DELETED

逻辑终止状态。

P2 不执行物理删除。

---

# 六、组织所有权

每个 TEAM 组织必须且只能有一个所有者。

所有者拥有：

```text
组织最高管理权
成员管理权
角色管理权
所有权转移权
组织解散权
```

所有权不应仅依赖普通角色。

原因：

```text
角色可以被修改
角色权限可以变化
所有者必须始终可识别
组织不能出现零所有者
组织不能出现多个模糊所有者
```

建议在：

```text
identity_organization.owner_user_id
```

保存当前所有者。

同时必须保证该用户存在有效 Membership。

业务不变量：

```text
owner_user_id 对应一个 ACTIVE Membership
所有者 Membership 不允许被移除
所有者不允许直接退出组织
转移所有权后，旧所有者可以继续保留 ADMIN 或 MEMBER 角色
```

所有权转移必须在同一个事务完成。

---

# 七、成员关系模型

## 7.1 Membership 状态

```text
ACTIVE
SUSPENDED
LEFT
REMOVED
```

### ACTIVE

正常成员。

### SUSPENDED

成员暂时被组织管理员停用。

效果：

```text
仍保留历史成员关系
不能访问组织资源
不能使用组织权限
可以由管理员恢复
```

### LEFT

成员主动退出。

### REMOVED

组织管理员移除成员。

`LEFT` 和 `REMOVED` 不物理删除，便于审计。

---

## 7.2 成员来源

记录成员如何加入组织：

```text
OWNER_CREATED
INVITATION
ADMIN_ADDED
SYSTEM_PROVISIONED
```

P2 实际使用：

```text
OWNER_CREATED
INVITATION
ADMIN_ADDED
```

`SYSTEM_PROVISIONED` 为后续 SCIM 预留。

---

## 7.3 成员不可变标识

Membership 必须拥有独立 ID。

不要只用：

```text
organization_id + user_id
```

作为所有业务引用。

原因：

```text
成员可能离开后重新加入
历史邀请与审计需要指向某次成员关系
角色分配需要稳定 ID
后续访问审查需要引用 Membership
```

可以保留：

```text
UNIQUE(organization_id, user_id)
```

表示 P2 暂时复用历史 Membership 记录，但外部引用仍使用 `membership_id`。

# 八、邀请模型

## 8.1 邀请流程

```text
组织管理员输入邮箱
    │
    ▼
选择初始角色
    │
    ▼
POST 创建邀请
    │
    ▼
检查组织状态与权限
    │
    ▼
检查邮箱是否已经是成员
    │
    ▼
创建 Invitation
    │
    ▼
生成一次性邀请 Token
    │
    ▼
写入 Outbox
    │
    ▼
core-notification 发送邮件
```

被邀请人点击链接后：

```text
未登录
  → 先登录或注册

已登录且邮箱匹配
  → 展示邀请详情

用户确认加入
  → 创建或恢复 Membership
  → 分配初始角色
  → Invitation 标记 ACCEPTED
```

---

## 8.2 邀请状态

```text
PENDING
ACCEPTED
DECLINED
REVOKED
EXPIRED
```

---

## 8.3 邀请规则

必须保证：

```text
邀请绑定目标组织
邀请绑定目标邮箱
邀请绑定发起人
邀请具有有效期
邀请 Token 只保存 Hash
邀请只能使用一次
撤销后不可再次接受
```

登录用户邮箱必须与邀请邮箱匹配。

P2 不允许：

```text
任何拿到邀请链接的人加入组织
```

---

## 8.4 已有账号和新账号

### 邮箱已经注册

用户登录后接受邀请。

### 邮箱尚未注册

用户先完成：

```text
注册
邮箱验证
```

验证邮箱必须与邀请邮箱一致。

注册完成后返回邀请确认页面。

---

## 8.5 重复邀请

同一组织、同一邮箱存在有效 PENDING 邀请时：

```text
不创建新记录
更新邀请有效期或重新发送原邀请
```

是否重置 Token 建议：

```text
重新发送时生成新 Token
旧 Token 立即失效
```

# 九、权限目录设计

权限不是组织用户随意输入的字符串。

所有可分配权限必须来自统一权限目录。

示例：

```text
identity.organization.read
identity.organization.update
identity.organization.delete

identity.member.read
identity.member.invite
identity.member.update
identity.member.remove

identity.role.read
identity.role.create
identity.role.update
identity.role.delete
```

其他 Core 也可以注册：

```text
storage.object.read
storage.object.create
storage.object.delete

billing.subscription.read
billing.subscription.manage

ai.model.use
ai.provider.manage
```

---

## 9.1 权限结构

每个权限包含：

```text
code
service
resource
action
name
description
risk_level
assignable
status
```

例如：

```text
code:
storage.object.delete

service:
core-storage

resource:
object

action:
delete

risk_level:
HIGH
```

---

## 9.2 权限命名规则

统一：

```text
{domain}.{resource}.{action}
```

例如：

```text
identity.member.invite
identity.role.manage
storage.object.read
billing.invoice.read
workflow.definition.execute
```

禁止：

```text
admin
super
manage_all
userPermission1
storage.*
```

P2 不支持通配权限。

一个权限码表达一个稳定能力。

---

## 9.3 权限风险等级

```text
LOW
MEDIUM
HIGH
CRITICAL
```

用途：

```text
前端危险提示
默认角色设计
高危权限过滤
后续访问审查
后续审批流
```

P2 不根据风险等级自动拒绝，只用于分类与展示。

---

# 十、权限如何由其他 Core 注册

Identity 不能手工维护整个生态的所有权限。

建议每个 Core 提供权限清单：

```yaml
service: core-storage
version: 1

permissions:
  - code: storage.object.read
    name: Read objects
    resource: object
    action: read
    riskLevel: LOW

  - code: storage.object.delete
    name: Delete objects
    resource: object
    action: delete
    riskLevel: HIGH
```

注册方式可以支持两种。

## 10.1 启动时内部 API 注册

其他 Core 启动后调用：

```text
PUT /internal/v1/identity/permission-sources/{service}
```

提交自身权限目录。

## 10.2 发布时同步脚本

由部署脚本调用 Identity Internal API。

P2 推荐优先使用：

```text
显式同步脚本
```

原因：

```text
避免 Identity 不可用导致业务服务无法启动
避免每次启动重复写权限
权限变化更容易审计
```

---

## 10.3 权限同步规则

新的权限：

```text
创建为 ACTIVE
```

修改名称和描述：

```text
允许更新
```

删除权限：

```text
不能直接物理删除
标记为 DEPRECATED
```

已经被角色使用的权限，不得直接移除。

最终删除必须经过：

```text
停用
角色引用清理
兼容期
```

---

# 十一、角色模型

## 11.1 角色类型

```text
BUILT_IN
CUSTOM
```

### BUILT_IN

系统为每个 TEAM 组织创建：

```text
OWNER
ADMIN
MEMBER
VIEWER
```

### CUSTOM

组织管理员自行创建。

---

## 11.2 内置角色

### OWNER

组织所有者身份。

实际所有权仍由：

```text
organization.owner_user_id
```

确定。

OWNER 角色用于权限展示和统一计算，但不能单独决定所有者身份。

### ADMIN

默认拥有：

```text
组织资料管理
成员邀请
成员调整
角色管理
大部分业务管理权限
```

不能：

```text
转移所有权
解散组织
修改所有者
```

除非后续明确授权。

### MEMBER

默认业务使用权限。

### VIEWER

默认只读权限。

---

## 11.3 内置角色可修改范围

建议：

```text
OWNER
不可删除
权限自动包含组织所有权核心权限

ADMIN
不可删除
允许调整部分业务权限

MEMBER
不可删除
允许调整业务权限

VIEWER
不可删除
允许调整只读权限
```

另一种更严格的方案是完全锁定内置角色。

P2 推荐折中：

```text
身份核心权限锁定
其他 Core 业务权限允许调整
```

例如 ADMIN 必须始终拥有：

```text
identity.organization.read
identity.member.read
identity.member.invite
identity.role.read
```

不能被组织管理员移除。

---

## 11.4 自定义角色

字段：

```text
名称
描述
权限集合
角色颜色或图标，可选
```

限制：

```text
同一组织内角色名称唯一
不能使用保留名称
不能创建空名称
不能包含已停用权限
```

是否允许空权限角色：

```text
允许
```

用途可能是先创建角色，再逐步配置。

但保存时应提示：

```text
该角色当前不授予任何权限
```

---

# 十二、成员角色分配

一个 Membership 可以拥有多个 Role。

例如：

```text
Ayşe
  ├── MEMBER
  └── FINANCE_MANAGER
```

最终权限取并集：

```text
Effective Permissions =
    Role A Permissions
    UNION
    Role B Permissions
```

P2 不支持显式 Deny，因此不会出现复杂优先级。

---

## 12.1 最少角色规则

每个 ACTIVE TEAM Membership 至少应有一个角色。

接受邀请时必须分配初始角色。

移除角色时，如果将导致成员零角色：

```text
拒绝操作
```

所有者始终拥有 OWNER 权限。

---

## 12.2 角色删除

删除自定义角色前必须处理：

```text
仍有哪些成员使用该角色
```

可选操作：

```text
将成员迁移到另一个角色
移除该角色并保留成员其他角色
取消删除
```

不能静默删除角色并让部分成员失去全部权限。

---

# 十三、当前组织上下文

一个用户可以属于多个组织，因此登录身份本身不足以完成授权。

每个组织级请求必须具有：

```text
current_organization_id
```

---

## 13.1 前端选择组织

用户侧顶部显示组织切换器。

当前选择保存在：

```text
前端本地偏好
```

同时后端 Session 可以记录：

```text
last_organization_id
```

用于下次登录恢复。

本地偏好只能用于 UX，不能作为授权事实。

---

## 13.2 请求传递

建议使用请求头：

```text
X-Organization-Id: organization-id
```

或者资源路径：

```text
/api/v1/identity/organizations/{organizationId}/members
```

P2 推荐：

```text
资源所属明确的接口使用 Path 参数
跨资源通用接口可以使用 X-Organization-Id
```

不要所有接口都隐式依赖 Header。

---

## 13.3 后端验证

每次组织级请求必须验证：

```text
用户 Session 有效
组织存在
组织状态允许当前操作
Membership 存在
Membership 状态为 ACTIVE
拥有要求的 Permission
```

禁止只验证：

```text
前端传来的 organizationId
```

---

## 13.4 当前组织失效

可能出现：

```text
用户已被移出当前组织
组织已被冻结
组织已删除
Membership 被暂停
```

前端收到组织上下文失效错误后：

```text
清除当前组织
重新加载可用组织
如果仍有组织，切换到第一个可用组织
如果没有组织，进入个人空间
```

不能无限重试原请求。

---

# 十四、授权服务设计

Identity Backend 内部建立：

```text
AuthorizationService
PermissionEvaluationService
OrganizationContextService
```

核心接口：

```java
public interface AuthorizationService {

    AuthorizationResult check(
        String userId,
        String organizationId,
        String permissionCode
    );
}
```

返回：

```text
ALLOW
DENY_USER_INACTIVE
DENY_ORGANIZATION_INACTIVE
DENY_MEMBERSHIP_NOT_FOUND
DENY_MEMBERSHIP_INACTIVE
DENY_PERMISSION_MISSING
```

公开响应不能泄露所有内部判断细节。

---

## 14.1 应用服务授权

组织级应用服务必须在入口检查权限。

例如：

```java
public void inviteMember(InviteMemberCommand command) {
    authorizationService.require(
        command.operatorUserId(),
        command.organizationId(),
        "identity.member.invite"
    );

    // 执行业务
}
```

禁止只在 Controller 层检查。

原因：

```text
Internal API
后台任务
未来 Workflow
```

也可能调用应用服务。

---

## 14.2 Controller 授权

Controller 可以执行快速入口保护，但不能替代 Application 层。

正确：

```text
API 层：
检查已登录、格式、基础 Scope

Application 层：
检查组织 Membership 和业务权限
```

---

# 十五、权限快照与性能

P2 使用 SQLite，不引入 Redis。

权限计算可以通过：

```text
Membership
→ MembershipRole
→ RolePermission
```

查询得到。

为降低重复查询，可以使用：

```text
单进程短期内存缓存
```

缓存 Key：


# 二十六、组织解散

P2 可以提供基础解散能力，但采用冷静期。

流程：

```text
所有者进入危险区域
    │
    ▼
查看影响
    │
    ▼
输入组织名称
    │
    ▼
验证当前密码
    │
    ▼
组织进入 PENDING_DELETION
    │
    ▼
通知其他 Core
    │
    ▼
冷静期后进入 DELETED
```

P2 不负责删除其他 Core 的数据。

各 Core 根据事件处理：

```text
冻结资源
阻止新增
执行自身数据保留策略
```

组织删除是跨服务流程，不能在 Identity 一个事务中完成。

## 30.3 不建议每次请求远程鉴权

P2 初期可以使用 Identity 权限检查 API。

但长期每次业务请求都远程调用 Identity 会形成强耦合。

因此 P2 同时提供两种模式：

### 模式一：远程鉴权

适合：

```text

低调用量
高风险管理操作
```

### 模式二：权限快照

业务 Core 获取当前用户在组织中的权限快照，在短时间内本地判断。

P3 Token 体系稳定后，再把权限或 Scope 放入标准授权令牌。

# 五十八、P1 到 P2 数据迁移

P1 已有：

```text
PERSONAL Organization
OWNER Membership
membership_type
```

P2 迁移步骤：

```text
1. 创建 Permission 表
2. 创建 Role 表
3. 创建 RolePermission
4. 创建 MembershipRole
5. 为每个现有个人空间创建 OWNER 内置角色
6. 将 P1 OWNER Membership 绑定 OWNER Role
7. 回填 organization.owner_user_id
8. 验证所有个人空间恰好有一个 Owner
9. 停止读取 membership_type
10. 后续迁移删除 membership_type
```

不要在一次生产升级中直接删除旧字段。

建议：

```text
P2.0 停止使用
P2.1 确认数据稳定
P2.2 再删除
```

# 六十一、P2 最重要的注意点

## 1. 不要把角色直接存在 User 表

错误：

```text
user.role = ADMIN
```

同一个用户在不同组织中可能是：

```text
A 组织：OWNER
B 组织：MEMBER
C 组织：VIEWER
```

角色必须通过 Membership 生效。

---

## 2. 不要让 Role 本身表示所有权

所有权是组织核心事实。

Role 是权限集合。

二者相关，但不能混为同一个字段。

---

## 3. 不要让其他 Core 自己维护组织成员

Storage、Billing、AI Gateway 可以保存：

```text
organization_id
created_by
```

但不能复制：

```text
organization_member
organization_role
organization_permission
```

Identity 是唯一成员与权限来源。

---

## 4. 不要把所有权限都写死在 Identity 源码

Identity 管理权限目录，但权限由各能力服务声明。

否则每增加一个 Storage 权限，都要发布 Identity 新版本。

---

## 5. 不要在 P2 引入 Deny

Allow 与 Deny 混用会产生：

```text
角色 A 允许
角色 B 拒绝
哪个优先
继承如何处理
```

P2 只做权限并集。

---

## 6. 不要把平台管理员和组织管理员混为一谈

组织管理员：

```text
管理自己的组织
使用 core-identity-web
```

平台管理员：

```text
治理全平台组织
使用 core-identity-admin-web
```

---

## 7. 不要远程鉴权每一个高频请求

P2 可以先提供远程检查，但同时设计权限快照和版本。

否则所有 Core 会对 Identity 形成高频同步依赖。

---

# 六十二、P2 最终成果

P2 完成后：

```text
core-identity-backend

成为用户、组织、成员、角色和权限的
唯一事实与规则中心。
```

```text
core-identity-web

成为用户进行组织切换、邀请成员、
管理角色和组织设置的自助门户。
```

```text
core-identity-admin-backend

成为平台跨组织治理的管理编排层，
但不拥有任何组织领域数据。
```

```text
core-identity-admin-web

成为平台查看组织、冻结风险组织、
审查成员和角色状态的管理控制台。
```

P2 最终建立的是一条完整授权链：

```text
用户
→ 当前组织
→ 成员关系
→ 角色
→ 权限
→ 具体操作
```

从这一阶段开始，Billing、Storage、AI Gateway、Workflow 等所有 Core 都可以围绕统一的：

```text
user_id
organization_id
permission_code
```

建立自己的业务能力，而无需重复建设成员与权限系统。
