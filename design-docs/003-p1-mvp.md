# Core Identity P1：Identity MVP 详细设计

版本：P1
前置依赖：P0 工程与边界基础完成
目标：交付第一版真正可被上层产品使用的身份服务。

---

# 一、P1 核心目标

P1 要让 `core-identity` 从工程骨架变成真正可用的身份系统。

必须完成：

```text
用户注册
邮箱验证
密码登录
安全退出
当前用户查询
个人资料维护
修改密码
忘记密码
密码重置
会话管理
最小组织模型
平台管理员登录
平台用户管理
账号状态治理
登录与操作审计
```

P1 完成后，其他 Core 至少能够回答：

```text
当前请求是谁发起的？
这个用户是否有效？
这个用户属于哪个个人空间？
这个账号是否已经被禁用？
```

---

# 二、P1 暂时不做什么

P1 不实现：

```text
完整组织管理
团队组织创建
成员邀请
复杂 RBAC
自定义角色
API Key
Service Account
JWT 对外授权体系
MFA
Passkey
第三方登录
OIDC
SAML
SCIM
企业 SSO
访问审批
企业合规治理
```

这些能力分别放在后续阶段。

P1 必须克制，避免把 MVP 做成未完成的企业 IAM。

# 四、P1 领域模型

P1 的核心领域关系：

```text
User
 ├── UserEmail
 ├── Credential
 ├── Session
 ├── PlatformOperator
 └── Membership
          │
          ▼
     Organization
```

注册用户时，不只创建一个 `User`。

必须在同一个事务中创建：

```text
User
UserEmail
Credential
Personal Organization
Owner Membership
Email Verification Token
Audit Event
Outbox Event
```

---

# 五、为什么 P1 必须创建个人组织

虽然 P1 不做完整组织管理，但注册时必须建立个人空间。

例如：

```text
用户：Ayşe
个人空间：Ayşe's Workspace
成员关系：OWNER
```

原因是后续 Core 都需要资源归属：

```text
Billing 的订阅属于哪个主体？
Storage 的文件属于哪个主体？
AI Gateway 的额度属于哪个主体？
Workflow 的任务属于哪个主体？
```

如果 P1 只有 `user_id`，P2 再强行增加组织，会导致大量资源归属迁移。

因此 P1 建立：

```text
organization_type = PERSONAL
```

P2 再增加：

```text
organization_type = TEAM
```

# 六、用户状态设计

## 6.1 User 状态

```text
PENDING_VERIFICATION
ACTIVE
LOCKED
DISABLED
```

状态含义：

### PENDING_VERIFICATION

```text
已经注册
邮箱尚未验证
不能正常进入账户
可以重新发送验证邮件
```

### ACTIVE

```text
邮箱已验证
可以正常登录
```

### LOCKED

```text
因连续登录失败等安全原因临时锁定
达到解锁时间后可以自动恢复
```

### DISABLED

```text
平台管理员主动禁用
不能自动恢复
必须由管理员启用
```

P1 不实现真正的数据删除。

账号删除和匿名化放在企业治理阶段。

---

## 6.2 Email 状态

```text
UNVERIFIED
VERIFIED
```

P1 一个用户只允许一个主邮箱，但数据结构允许未来扩展多个邮箱。

---

## 6.3 Credential 状态

```text
ACTIVE
MUST_CHANGE
DISABLED
```

其中：

```text
ACTIVE
正常凭证

MUST_CHANGE
管理员创建账号或安全事件后，下一次登录必须修改密码

DISABLED
凭证不可用于登录
```

---

## 6.4 Session 状态

```text
ACTIVE
REVOKED
EXPIRED
```

数据库中不一定需要持续写入 `EXPIRED`，可以根据过期时间动态计算。

# 七、用户注册流程

## 7.1 主流程

```text
用户打开注册页
    │
    ▼
填写邮箱、密码、显示名称
    │
    ▼
前端本地基础校验
    │
    ▼
POST /api/v1/identity/auth/register
    │
    ▼
后端检查邮箱是否已经存在
    │
    ▼
创建用户及个人组织
    │
    ▼
生成邮箱验证 Token
    │
    ▼
写入审计和 Outbox
    │
    ▼
调用 core-notification
    │
    ▼
跳转邮箱待验证页
```

---

## 7.2 注册事务

以下操作必须在同一个数据库事务中完成：

```text
创建 User
创建 UserEmail
创建 Credential
创建 Personal Organization
创建 OWNER Membership
创建 OneTimeToken
写入 Audit Event
写入 Outbox Event
```

通知发送不能放在数据库事务中。

正确流程：

```text
数据库事务提交
    │
    ▼
Outbox 或通知任务
    │
    ▼
core-notification
```

避免邮箱服务失败导致用户注册事务回滚。

---

## 7.3 注册幂等

注册接口接受：

```text
Idempotency-Key
```

同一个幂等键重复提交时：

```text
请求内容相同
    → 返回第一次结果

请求内容不同
    → 返回 409 IDEMPOTENCY_CONFLICT
```

邮箱唯一约束是最终保障，幂等记录用于改善重复点击和网络重试体验。

---

## 7.4 邮箱已存在时的响应

公开注册接口不应明确暴露某个邮箱是否已注册。

建议统一返回：

```text
如果该邮箱可以注册，我们已经发送后续指引。
```

但用户体验上容易困惑。

P1 可以采用折中方式：

```text
如果邮箱已注册并已激活：
    提示“该邮箱无法用于注册，可以尝试登录或找回密码”

如果邮箱已注册但未验证：
    提示“验证邮件已重新发送”
```

必须配合限流，防止被用于批量枚举邮箱。

## 8.1 页面字段

```text

```

```text

```

身份服务不应在注册阶段收集不必要信息。

---

## 8.2 密码输入体验

必须提供：

```text
显示或隐藏密码
密码规则实时反馈
Caps Lock 提示
粘贴支持
密码管理器支持
```

禁止：

```text

```

密码规则在前端用于提示，在后端用于最终校验。

---

## 8.3 提交状态

提交后按钮进入：

```text
正在创建账户……
```

提交期间：

```text
按钮禁用
避免重复提交
输入内容保留
```

失败时：

```text
保留非密码字段
清空密码字段
聚焦第一个错误字段
展示 Request ID
```

---

## 8.4 注册成功页

注册成功后不直接进入账户。

进入：

```text
/check-email
```

展示：

```text
验证邮件发送到哪个邮箱
重新发送按钮
修改邮箱入口
邮箱客户端快捷入口
剩余冷却时间
```

示例：

```text
验证邮件已经发送到 a***@example.com。

请在 30 分钟内完成验证。
```

不要完整显示邮箱，避免公共设备泄露信息。

# 九、邮箱验证流程

## 9.1 Token 设计

邮件中携带一次性随机 Token：

```text
https://identity.example.com/verify-email?token=...
```

数据库只保存：

```text
token_hash
```

不保存明文 Token。

Token 必须：

```text
一次性使用
有限有效期
使用后立即失效
用户状态变更后可以整体作废
不能通过自增 ID 猜测
```

---

## 9.2 验证流程

```text
用户点击验证链接
    │
    ▼
GET 前端验证页面
    │
    ▼
POST /api/v1/identity/auth/email-verifications/confirm
    │
    ▼
后端检查 Token
    │
    ├── 有效
    │     ├── 邮箱标记 VERIFIED
    │     ├── 用户标记 ACTIVE
    │     ├── Token 标记 USED
    │     ├── 写入审计
    │     └── 发布事件
    │
    ├── 已使用
    │     └── 显示已经验证
    │
    ├── 已过期
    │     └── 提供重新发送
    │
    └── 无效
          └── 显示安全错误
```

---

## 9.3 验证成功 UX

显示：

```text
邮箱验证成功
账户已经激活
进入登录
```

不要在邮件验证后自动创建长期登录会话。

原因：

```text
链接可能在邮件安全扫描器中被提前打开
链接可能在其他设备打开
```

验证成功后让用户正常登录，更安全也更容易解释。

---

# 十、用户登录设计

## 10.1 P1 登录方式

只支持：

```text
邮箱 + 密码
```

暂时不支持：

```text
用户名登录
手机号登录
验证码登录
第三方登录
Passkey
```

登录入口越多，账号合并和安全边界越复杂。

---

## 10.2 登录流程

```text
用户输入邮箱和密码
    │
    ▼
POST /api/v1/identity/auth/login
    │
    ▼
标准化邮箱
    │
    ▼
检查 IP 和账号登录限制
    │
    ▼
查询用户与凭证
    │
    ▼
验证密码
    │
    ├── 失败
    │     ├── 记录 LoginAttempt
    │     ├── 增加失败计数
    │     ├── 必要时临时锁定
    │     └── 返回统一错误
    │
    └── 成功
          ├── 检查用户状态
          ├── 创建 Session
          ├── 记录 LoginAttempt
          ├── 写入审计
          ├── 设置 Cookie
          └── 返回当前用户摘要
```

---

## 10.3 登录失败提示

以下情况不应分别暴露：

```text
邮箱不存在
密码错误
凭证不存在
```

统一返回：

```text
邮箱或密码不正确。
```

以下状态可以明确提示：

```text
邮箱尚未验证
账号已被禁用
账号临时锁定
```

但不要披露内部安全细节。

---

## 10.4 登录页面 UX

页面字段：

```text
邮箱
密码
记住此设备
```

P1 可以暂时不实现长期“记住我”。

如果保留该控件，必须真正改变 Session 的有效期，不能只是装饰。

页面操作：

```text
登录
忘记密码
创建账户
重新发送验证邮件
```

登录失败后：

```text
保留邮箱
清空密码
聚焦密码框
显示错误原因
显示 Request ID
```

# 十一、会话设计

P1 浏览器登录使用：

```text
Opaque Session Token
```

而不是对浏览器直接使用长期 JWT。

原因：

```text
可以立即撤销
账号禁用后立即失效
密码重置后可以撤销全部会话
安全行为容易理解
适合单节点 SQLite MVP
```

P3 再增加面向其他 Core 和外部程序的标准 Token 体系。

---

## 11.1 Cookie

用户侧 Cookie：

```text
core_identity_session
```

管理侧 Cookie：

```text
core_identity_admin_session
```

必须设置：

```text
HttpOnly
Secure（生产环境）
SameSite
合理的 Path
```

用户侧和管理侧 Cookie 不得共用名称。

---

## 11.2 Session Token

浏览器持有：

```text
随机高熵 Session Token
```

数据库只保存：

```text
session_token_hash
```

禁止明文保存完整 Session Token。

---

## 11.3 会话时间

建议同时包含：

```text
idle_expires_at
absolute_expires_at
```

例如：

```text
长时间无操作后失效
即使持续操作，也不能超过绝对最长时间
```

具体时间通过配置决定，不写死在业务代码中。

---

## 11.4 会话管理页面

用户可以看到：

```text
设备名称
浏览器
操作系统
大致 IP
创建时间
最近活动时间
当前设备标记
```

用户可以：

```text
退出单个其他会话
退出所有其他会话
```

当前会话退出后跳转登录页。

# 十三、修改密码

## 13.1 流程

```text
用户输入当前密码
用户输入新密码
用户确认新密码
    │
    ▼
后端验证当前密码
    │
    ▼
检查新旧密码是否相同
    │
    ▼
验证密码规则
    │
    ▼
更新 Credential
    │
    ▼
撤销其他会话
    │
    ▼
写入审计
    │
    ▼
发送安全通知
```

默认策略：

```text
保留当前会话
撤销其他所有会话
```

也可以配置为全部撤销。

---

## 13.2 UX

修改成功后明确提示：

```text
密码已经更新。

为了保护账户，其他设备上的登录已经退出。
```

不能只显示：

```text
Success
```

# 十四、忘记密码与密码重置

## 14.1 发起重置

```text
POST /api/v1/identity/auth/password-resets
```

无论邮箱是否存在，公开响应都相同：

```text
如果该邮箱对应有效账户，我们已经发送密码重置邮件。
```

防止批量枚举用户。

---

## 14.2 重置 Token

使用 `identity_one_time_token`。

Token 类型：

```text
PASSWORD_RESET
```

要求：

```text
只保存 Hash
一次性使用
有限有效期
发出新 Token 时废弃旧 Token
密码修改后全部失效
```

---

## 14.3 完成重置

```text
用户点击邮件链接
    │
    ▼
输入新密码
    │
    ▼
后端验证 Token
    │
    ▼
更新 Credential
    │
    ▼
撤销全部 Session
    │
    ▼
Token 标记 USED
    │
    ▼
写入审计和安全事件
    │
    ▼
发送密码已重置通知
```

密码重置完成后必须重新登录。

# 十五、管理员身份设计

P1 需要平台管理员，但不提前建设完整平台 RBAC。

建立：

```text
PlatformOperator
```

P1 只有：

```text
SUPER_ADMIN
```

普通用户与平台管理员不是两套 User。

平台管理员仍然是：

```text
identity_user
```

只是额外拥有：

```text
identity_platform_operator
```

记录。

这样可以复用：

```text
邮箱
密码
会话
账号状态
安全规则
审计规则
```

# 十六、首个管理员初始化

禁止默认创建：

```text
admin / admin123
```

推荐提供初始化命令：

```text
scripts/bootstrap-admin.bat
scripts/bootstrap-admin.sh
```

输入：

```text
管理员邮箱
显示名称
```

系统执行：

```text
创建或查找 User
创建 PlatformOperator
生成一次性账户设置 Token
```

```text

```

初始化完成后，Bootstrap 能力自动关闭。

# 十七、管理员登录流程

```text
Admin Web
    │
    │ POST /admin-api/v1/identity/auth/login
    ▼
Admin Backend
    │
    │ Service Token + 管理登录请求
    ▼
Identity Backend Internal API
    │
    ├── 验证邮箱和密码
    ├── 验证 PlatformOperator
    ├── 验证 User 状态
    ├── 创建 ADMIN_WEB Session
    └── 返回一次性 Session Token
    │
    ▼
Admin Backend
    │
    ├── 设置 Admin HttpOnly Cookie
    └── 返回管理员摘要
```

Admin Backend 不保存或验证密码。

Identity Backend 是唯一密码验证者。

# 三十五、密码安全设计

P1 基础要求：

```text

密码规则后端强制执行
```

# 五十五、P1 最重要的注意点

## 1. 不要用 User 表兼任一切

不要往 `identity_user` 塞入：

```text
email
password
organization_name
role
is_admin
subscription
quota
```

应分别由：

```text
UserEmail
Credential
Organization
Membership
PlatformOperator
Billing
```

负责。

---

## 2. 不要让管理员设置用户永久密码

管理员只能：

```text
创建用户
发送账户设置链接
触发密码重置
```

不能看到或保存用户密码。

---

## 3. 不要在 P1 提前设计复杂角色

P1 只有：

```text
个人空间 OWNER
平台 SUPER_ADMIN
```

完整组织角色与权限放到 P2。

---

## 4. 不要让通知失败破坏注册事务

注册成功与邮件发送成功是两个不同事实。

正确体验：

```text
账户已创建
验证邮件暂时发送失败
用户可以稍后重新发送
```

而不是回滚整个用户。

---

## 5. 不要给浏览器 Session 使用不可撤销的长期 JWT

P1 首先需要：

```text
可靠注销
管理员强制退出
密码重置后立即失效
账号禁用后立即失效
```

Opaque Session 更适合 MVP。

---

## 6. 不要把个人空间展示成复杂企业组织

P1 用户只需要感知：

```text
我的空间
```

不需要在界面中解释：

```text
Tenant
Membership
Organization Type
RBAC
```

底层模型可以完整，用户体验必须简单。

---

## 7. 管理员与组织管理员不是一回事

P1 的平台管理员使用：

```text
core-identity-admin-web
```

未来组织管理员使用：

```text
core-identity-web
```

二者权限和安全边界完全不同。
